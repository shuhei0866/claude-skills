# /delegate - タスク委譲フレームワーク

**Announce at start:** "委譲フレームワークに基づいてタスクをルーティングします。"

## 概要

このスキルは、オーケストレーター（メイン会話の Claude）が以下を判断するためのガイドラインを提供する:

1. タスクを**どのモデル/プロバイダ**に委譲するか
2. **どのエージェント**を使い、**どのスキル**をプリロードするか
3. 完了後に**品質メトリクスをどう記録**するか

## Step 1: タスク分類

受け取ったタスクを以下のカテゴリに分類する。

| タスク種別 | 判定基準 |
|-----------|---------|
| `complex-implementation` | 複数ファイルにまたがる、設計判断が必要、ツール連携が多い |
| `standard-implementation` | 単一機能の追加・修正、パターンに従う実装 |
| `code-review` | 既存コードの品質チェック、PR レビュー |
| `test-writing` | テストケースの設計・実装 |
| `codebase-research` | 既存コードの調査・理解、ファイル探索 |
| `web-research` | Web 上の情報収集、ドキュメント調査 |
| `social-research` | X/Twitter、SNS の情報収集 |
| `documentation` | ドキュメント執筆・更新 |
| `bug-investigation` | バグの根本原因分析、デバッグ |
| `planning` | 設計・プラン策定、アーキテクチャ判断 |

### 複雑度の判定

| 複雑度 | 基準 |
|--------|------|
| **高** | 5ファイル以上の変更、アーキテクチャ判断、セキュリティ/認証関連、DB スキーマ変更 |
| **中** | 2-4ファイルの変更、既存パターンの応用、テスト作成 |
| **低** | 1ファイルの変更、定型作業、検索・調査 |

## Step 2: モデル選択

[model-registry.md](references/model-registry.md) を参照して利用可能なモデルを確認し、
[routing-table.md](references/routing-table.md) に基づいてモデルを選択する。

### 選択の原則

1. **コスト最適化**: タスクに十分な能力を持つ最も安いモデルを選ぶ
2. **リスク考慮**: 手戻りコストが高いタスクは上位モデルを選ぶ
3. **ツール依存**: Claude Code のツール連携が必要なら Claude 系モデル、純粋なコードなら Codex も可

## Step 3: エージェント選択とスキルプリロード

### Claude サブエージェント委譲時

```
Task tool 呼び出し:
  subagent_type: <エージェント名>
  model: <routing-table で決定したモデル>
  prompt: <タスク内容>
```

**エージェント別のスキル紐付け:**

| エージェント | プリロードスキル | 用途 |
|------------|-------------|------|
| `implementer` | db-migration, env-secrets | 実装タスク全般 |
| `code-reviewer` | security-balance | コードレビュー、品質チェック |
| `release-manager` | release | リリース管理 |
| `debugger` | (env-secrets は vercel-debug 経由) | バグ調査 |
| `metrics-recorder` | delegate | メトリクス記録 |

**Task tool の model パラメータ**で委譲先モデルを制御する（エージェント定義は `inherit` のまま）。

### 外部プロバイダ委譲時

```bash
# Codex で実装
codex "タスク内容"

# Codex でレビュー
codex review --base <branch>

# Gemini CLI（将来）
gemini "タスク内容"

# Grok CLI（将来）
grok "タスク内容"
```

外部 CLI は Bash tool 経由で実行する。ツール連携やファイル操作が不要な「純粋な思考・生成タスク」に適する。

## Step 4: 品質チェック

委譲先からの成果物に対し、以下を確認する:

### 自動チェック（実行可能な場合）

```bash
pnpm typecheck   # 型チェック
pnpm lint        # Lint
pnpm test:run    # テスト
pnpm build       # ビルド
```

### レビューチェック

- `/review-now` または `/codex-review` の指摘数・重大度
- スコープ遵守（リリース仕様書との差分）

### エスカレーション判定

品質チェックは2段階で行う:

1. **自動チェック後（即時）**: テスト・型チェック・ビルドの結果を判定
2. **レビュー後（Phase 3）**: レビュー指摘の重大度・件数で判定

| 段階 | 状況 | アクション |
|------|------|---------|
| 自動チェック後 | テスト/型チェック/ビルド失敗 | 同じモデルで1回リトライ → 失敗なら上位モデル |
| レビュー後 | レビューで Critical 指摘 | 上位モデルに再委譲 |
| レビュー後 | レビューで High 指摘 2件以上 | 上位モデルに再委譲 |
| レビュー後 | スコープ大幅逸脱 | 上位モデルに再委譲 |

**エスカレーション順序:** haiku → sonnet → opus → (人間判断)

## Step 5: メトリクス記録

タスク完了後、`metrics-recorder` エージェントを使って Notion DB に記録する。

**記録タイミング:** 各委譲タスクの完了時（成功・失敗問わず）

**記録方法:**
```
Task tool:
  subagent_type: metrics-recorder
  model: haiku
  prompt: |
    以下のメトリクスを Notion の Delegation Log に記録してください。
    [notion-schema.md](references/notion-schema.md) のスキーマに従ってください。

    - タスク種別: <種別>
    - 複雑度: <高/中/低>
    - 使用モデル: <モデル名>
    - テスト通過: <yes/no>
    - カバレッジ差分: <数値>
    - レビュー指摘: Critical <N>, High <N>, Medium <N>
    - 手戻り回数: <N>
    - スコープ遵守: <遵守/軽微逸脱/大幅逸脱>
    - トークン概算: <数値>
    - 所要時間(分): <数値>
    - リリース名: <release/xxx>
    - 備考: <自由記述>
```

## フィードバックループ

品質データが蓄積された後、以下のサイクルで委譲ルールを改善する:

1. **データ分析**: Notion DB のメトリクスを集計・分析
2. **パターン検出**: モデル別・タスク種別別の品質傾向を特定
3. **ルール変更提案**: AI が routing-table.md の更新案を作成
4. **人間承認**: 提案をユーザーに提示し、承認を得てから適用

このサイクルは手動トリガー（将来の `/delegation-review` スキルで自動化予定）。

## 注意事項

- このスキルは `context: fork` ではない。オーケストレーターの会話内でインライン参照する
- 外部 CLI（Codex 等）は Claude Code のツール連携が使えないため、ツール操作が必要なタスクには不向き
- Gemini / Grok は CLI 未導入。model-registry.md で「計画中」として管理
- メトリクス記録を省略しない。データがなければフィードバックループが回らない
